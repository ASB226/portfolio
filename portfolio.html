기술지원 & 장애 해결 사례 1

상황

Object Storage 기반 PV의 간헐적 연결 실패로 인한 Kubernetes 환경 서비스 중단 발생

  • 초기 증상: vSAN File Service 기반 PV 사용 Pod의 간헐적 연결 실패, 기존 정상 Pod도 스토리지 연결 해제되며 비정상 종료
  • 임시 조치: CSI Controller 재시작 시 일시 정상화되나, 일정 시간 경과 후 재발하는 패턴 반복
  • 환경: VMware 가상화 인프라 + vSAN 분산 스토리지 + Tanzu Kubernetes

원인 분석

주요 원인

  • 하이퍼바이저 소프트웨어 버그: VMware ESXi 7.0 Update 3f 버전 스토리지 객체 처리 로직 오류로 Unknown Object 무한 생성
  • vSAN 성능 저하: Unknown Object 누적으로 분산 스토리지 불안정, PV 연결 실패 발생
  • 패치 진행 불가: Unknown Object 대량 존재로 vSAN 사전 호환성 검증 실패, 버그 해결된 버전으로의 패치 작업 진행 불가

장애 확산 과정

  1. 패치 시도: 하이퍼바이저 버그 해결을 위한 업그레이드 진행
  2. 패치 불가: Unknown Object 다수 존재로 vSAN 사전 호환성 검증 실패
  3. Unknown Object 삭제 및 하이퍼바이저 패치 진행
  4. 패치 중 장애: VM 마이그레이션 실패, 호스트 응답 없음, 네트워크 성능 저하 등 복합 이슈 발생
  5. 장애 확산 및 서비스 중단: 패치중 Control Plane 1대·Worker Node 3대 손실 발생, 노드 자동 복구 기능이 트리거 실패하며 Tanzu Kubernetes 기능 hang 상태 지속
  6. Kubernetes 배포용 OS 이미지 손실 인식: Unknown Object 삭제 중 Kubernetes 배포용 OS 이미지를 관리하는 Content Library 저장소가 삭제되는 이슈 발생

→ vSAN 데이터스토어 접근 불가 및 Kubernetes 클러스터 접근을 포함한 모든 PV 사용 애플리케이션 동작 중단

해결 과정

1단계: 긴급 대응 및 자체 조치

  • 내부 가능 범위 우선 대응으로 복구 시간 단축
  • vSAN 관리 서비스(vsanmgmtd, osfsd) 재시작, vCenter vpxd 서비스 재부팅
  • 네트워크 레벨 이슈 해결 (NIC 성능 저하, vDS 포트 불일치)
  • CSI 드라이버 업그레이드 및 Supervisor 클러스터 레벨 트러블슈팅, 장애 영향 범위 파악

2단계: 벤더 협력 통한 시스템 복구

  • vSAN File Service 재구성: VMware 엔지니어 가이드로 기존 구성 삭제 후 재배포
  • Content Library 복구: 새 Kubernetes 노드 배포 이미지 업로드

3단계: 서비스 정상화 검증

  • Kubernetes 클러스터 재구성: 기존 노드 유지하며 롤링 업데이트 트리거
  • 신규 PV 생성 및 Pod 스토리지 연결 기능 복구 확인
  • 애플리케이션 정상 동작 테스트 완료

성과 및 예방책

성과

  • 서비스 복구: Kubernetes 서비스 및 애플리케이션, 스토리지 시스템 정상화
  • 데이터 보호: 대부분의 PV 데이터 손실 없이 복구 (일부 vSAN File Service 기반 RWX 볼륨 손실)

예방책

  • 스토리지 모니터링 강화: CSI 드라이버 상태 및 PV 연결 상태 알림 환경 구성
  • Unknown Object 관리 프로세스 수립: 안전한 식별 및 삭제 절차 문서화
  • 정기점검 항목 추가 및 근본 원인 분석: Unknown Object 모니터링 및 vSAN object 이슈 근본 원인 확인을 위한 벤더 분석 요청

기술지원 & 장애 해결 사례 2

상황

cert-manager 메모리 누수로 인한 인증서 갱신 실패

  • 증상: cert-manager 컨트롤러 재시작 현상 간헐적 발생, 일부 애플리케이션 인증서 Ready=False 상태 반환 및 HTTPS 접속 불가
  • 환경: VMware 가상화 인프라 + Tanzu Kubernetes + cert-manager v1.4

원인 분석

문제 인식 과정

  1. 정기점검 시 대시보드 모니터링을 통해 cert-manager Pod AGE가 1d로 재시작 된 현상 확인
  2. 메세지 및 로그 확인 후 2주간 Grafana 모니터링 결과 컨트롤러 메모리 사용량이 resource limit 한계치까지 지속 증가 후 Pod 재시작 패턴 반복
  3. 전체 서비스중 일부 CertificateRequest 실패로 Ready=False 상태 발생, 접속 불가 확인

주요 원인

  • cert-manager 메모리 누수: 컨트롤러 내부 타이머 goroutine 누수로 Pod 메모리 사용량 지속 증가 (cert-manager 공식 GitHub PR #4233 에서 확인)
  • 만료일 임박 인증서 갱신 실패: cert-manager 컨트롤러 이슈로 인해 일부 애플리케이션 인증서 갱신 작업이 제대로 실행되지 않음

해결 과정

1단계: 임시 조치

  • kubectl cert-manager renew 명령어를 활용한 인증서 수동 갱신 실행 → 갱신 성공 및 애플리케이션 정상 접속 확인
  • kubectl get certificates -A --field-selector status.conditions[0].status=False 명령어로 Ready=False 상태의 인증서 범위 식별
  • bash 스크립트를 활용해 전체 네임스페이스 범위에 만료된 인증서 추출, for 루프를 통한 일괄 갱신 진행

2단계: 원인 확인 및 조치

  • GitHub cert-manager 리포지토리 PR #4233 "Fix timer goroutine leak" goroutine 누수 이슈 확인
  • cert-manager v1.5 릴리즈 노트 Bug or Regression 항목에 Fix a goroutine leak that was causing the controller's memory usage to grow with time. Fixed in the cert-manager PR #4233 확인
  • cert-manager v1.4 to v1.5.2 업그레이드 계획 수립
  • 업그레이드 이후 모니터링 시 메모리 누수 현상 해소 확인

성과 및 예방책

성과

  • 서비스 정상화: 인증서 수동 갱신으로 빠른 임시 조치 및 업그레이드 이후 만료 임박한 인증서 갱신 프로세스 정상화 확인

예방책

  • 버전 관리: cert-manager와 kubenetes 버전 호환 유지 및 전체 오픈소스 대상 정기적으로 known issue 파악 후 업그레이드 하는 프로세스 정립
  • 정기점검 항목 추가: cert-manager 리소스 사용률 및 kubectl get certificates -A --field-selector status.conditions[0].status=True 명령어를 사용해 Ready=True 상태 반환 확인하는 항목 추가
